# Workloop 本地编码代理编排设计

- 日期：2026-07-10
- 状态：设计已完成 grilling，待拆分实施计划
- 目标版本：下一代核心执行链（第一版）
- 适用范围：本地 Git 项目、Claude Code、Codex CLI、Web 控制台

## Problem Statement

当前 Workloop 已经具备任务状态、上下文门禁、模型路由、工作区快照、代码审核、交付门禁和审计工件，但 AI 调用方式仍然建立在“向一个 CLI 进程传入大段提示词，再把标准输出解析为 JSON”的假设上。

这个假设不适合真实编码任务：

- Claude Code 和 Codex CLI 是具备代码搜索、文件编辑、命令执行、会话恢复和权限控制能力的编码代理，不只是文本模型入口。
- 当前执行者只得到文件名列表，没有直接读取真实代码的可靠能力，却被要求返回完整文件内容 JSON。
- 当前审核者主要看到计划和 diff，不能以只读方式独立检查完整工作区并复跑验证。
- 通用命令模板无法统一表达工作目录、读写权限、工具白名单、结构化输出、流式事件、会话、费用和超时策略。
- CLI 子进程没有形成可靠的任务级隔离、持久恢复和供应商协议适配边界。
- 当前“流程编排”界面允许移动、增删和禁用节点，但内核仍执行固定 planner、executor、reviewer 顺序，页面表达的能力超过实际能力。
- `done` 同时承担“审核通过”和“已经交付”的含义，失败继续、计划返工和审核返修的状态语义也不够精确。
- 经验记忆、任意流程配置、模型命令编辑和多任务并发在核心闭环可靠之前增加了不必要的产品与架构复杂度。

用户真正需要的不是一个通用模型路由器，也不是一个新的编码代理。用户需要把真实需求交给 Workloop，由 Claude Code 分析和制定计划，由 Codex CLI 编写代码，再由独立 Claude Code 会话审核；审核不通过时将具体建议交给 Codex 返修，直到审核通过或必须由人介入。Workloop 负责隔离、编排、确定性验证、状态恢复、审计和人工门禁。

## Solution

Workloop 将被收缩并重新定义为本地编码代理编排器。

用户选择一个已注册的 Git 项目和目标分支，提交自然语言需求。Workloop 从不可变基线提交创建独立任务 worktree，让 Claude Code 以只读权限直接分析代码并逐项澄清高影响问题。Claude 生成固定结构的 `ExecutionPlan`，用户批准后计划被冻结。

Codex CLI 在同一任务 worktree 中以受限写权限执行计划、直接编辑文件并运行开发期测试。Workloop 对执行前后的工作区生成快照和 diff，检查路径与权限策略，并在受控环境中独立运行计划要求的测试。随后，一个与规划会话隔离的 Claude Code 审核会话以只读权限检查完整工作区、计划、diff 和验证证据。

审核可以产生四种处置：

- `pass`：实现满足全部验收标准，可以生成交付报告。
- `revise_code`：实现存在问题，恢复原 Codex 会话进行返修。
- `replan`：已批准计划本身存在问题，返回 Claude 重新规划并再次人工批准。
- `blocked`：存在安全风险、需求歧义或无法可靠判断的问题，暂停等待人工处理。

执行与审核最多自动循环三轮。达到轮次、时间、费用或无事件预算后任务暂停，不静默切换代理或模型。审核通过后，Workloop 生成结构化 `DeliveryReport` 和可审计任务提交。用户确认交付前，Workloop 检查目标分支是否仍与任务基线兼容；目标分支前进时必须重新整合、验证和审核。

第一版使用 Claude Code 和 Codex CLI 专用适配器。模型可按角色配置，但任务启动后冻结。Workloop 的核心契约面向 `AgentRuntime`，未来可以增加 Anthropic 或 OpenAI API Runtime，但第一版不实现 API 工具循环。

Web 控制台是主要操作入口，CLI 是底层运行入口和自动化备用接口。第一版采用固定语义工作流，只配置“谁承担角色”，不提供任意节点编排。

## User Stories

1. 作为开发者，我希望注册一个本地 Git 项目，以便 Workloop 可以为后续任务建立可靠的项目边界。
2. 作为开发者，我希望为项目声明默认目标分支，以便每个任务都有明确的交付目标。
3. 作为开发者，我希望项目配置可以进入版本控制，以便团队能够审查测试命令、受保护路径和权限边界。
4. 作为开发者，我希望 Workloop 在创建任务前检查工作区是否干净，以免无关的未提交修改进入任务。
5. 作为开发者，我希望任务记录不可变的基线提交，以便所有 diff、审核和交付结论都可以复现。
6. 作为开发者，我希望只需要选择项目、目标分支并填写需求，以免手工维护上下文文件和播种目录。
7. 作为开发者，我希望可以为需求附加项目外资料，以便 Claude 能够使用规格、截图或补充文档。
8. 作为开发者，我希望 Workloop 自动创建独立任务 worktree，以免编码代理修改真实工作目录。
9. 作为开发者，我希望 Claude 在分析阶段只能读取任务工作区，以免计划阶段产生未审查的代码修改。
10. 作为开发者，我希望 Claude 自行搜索和打开相关代码，以免 Workloop 错误猜测需要拼接哪些文件。
11. 作为开发者，我希望 Claude 的分析引用具体文件和关键符号，以便我确认计划基于真实代码而不是泛化推测。
12. 作为开发者，我希望 Claude 对高影响歧义一次提出一个带推荐答案的问题，以便我能集中作出产品决定。
13. 作为开发者，我希望低风险实现细节由 Claude 在计划中明确选择，以免流程被不必要的问题打断。
14. 作为开发者，我希望存在未决问题时禁止进入执行阶段，以免 Codex 在错误假设下修改代码。
15. 作为开发者，我希望 Claude 输出结构化计划，以便同一份计划既可供我审阅，也可供 Codex 执行。
16. 作为开发者，我希望计划包含需求理解和非目标，以便防止任务范围在执行中扩张。
17. 作为开发者，我希望计划标明涉及文件和符号，以便检查预计的改动范围。
18. 作为开发者，我希望计划包含实施步骤、约束和风险，以便 Codex 能按明确边界执行。

## Implementation Decisions

### 1. 产品边界

- Workloop 是本地编码代理编排器，不是通用模型聊天界面、通用工作流引擎或新的编码代理。
- Workloop 负责项目隔离、任务状态、代理编排、权限翻译、确定性验证、工件记录、恢复和人工门禁。
- Claude Code 和 Codex CLI 负责在各自授权范围内理解代码、使用工具和完成角色工作。
- 第一版固定使用“Claude 规划、Codex 执行、Workloop 验证、Claude 审核”的语义工作流。
- 角色使用的 runtime 与 model 可配置，但任务启动后冻结。

### 2. 保留与替换边界

- 保留现有工件存储、路径策略、工作区快照与 diff、交付门禁及其已验证行为。
- 将现有上下文、审核和交付工件迁移到新状态机可以引用的版本化结构中。
- 替换当前一次性 `run_model_loop` 编排、字符串输入输出模型契约和通用可写 CLI 后端。
- 替换仅存在内存中的运行注册表，队列与运行状态必须持久化。
- 移除或收缩当前任意流程节点编辑能力，模型与流程页面改为运行时健康状态和角色绑定。
- 经验记忆保留历史数据，但退出第一版主流程，不自动提炼或注入。

### 3. 领域对象

- `Project`：已注册的本地 Git 项目，包含项目标识、仓库根、默认目标分支和项目策略引用。
- `Task`：一次从真实需求到可交付提交的持久工作单元。
- `TaskWorkspace`：从不可变基线提交创建的独立 worktree 与任务分支。
- `ExecutionPlan`：Claude 生成、用户批准并冻结的执行契约。
- `ExecutionResult`：Codex 对一次执行或返修的结构化总结。
- `ValidationRun`：Workloop 独立执行的一组确定性命令及证据。
- `ReviewResult`：Claude 对计划、代码和验证证据的结构化审核结果。
- `DeliveryReport`：最终审核通过后面向用户的完整交付结论。
- `AgentProfile`：角色可选择的 runtime、model、权限、预算和版本策略。
- `AgentSession`：供应商会话引用；它优化连续性，但不是任务恢复的唯一来源。
- `AgentRun`：一次具体代理调用，包含冻结输入、运行身份、事件流、预算和最终结果。

### 4. 持久状态机

任务状态采用显式阶段和等待原因，不再用单一 `done` 混合审核与交付语义。建议状态集合为：

- `draft`
- `preparing_workspace`
- `analyzing`
- `waiting_for_clarification`
- `planning`
- `waiting_for_plan_approval`
- `queued_for_execution`
- `executing`
- `validating`
- `reviewing`
- `queued_for_revision`
- `replanning`
- `blocked`
- `interrupted`
- `ready_to_deliver`
- `integrating`
- `delivered`
- `failed`
- `cancelled`

状态迁移必须通过单一状态机服务完成，并校验允许的前置状态。每次迁移记录原因、触发者、关联工件、时间和下一步动作。

每个长阶段遵守“先写运行记录，再执行外部动作，最后原子提交结果”的顺序。服务启动时扫描未终结 `AgentRun` 与 `ValidationRun`，将无法确认仍在运行的阶段标记为 `interrupted`，不自动假定成功或失败。

### 5. 固定工作流

标准路径为：

1. 注册项目并确认干净基线。
2. 创建任务 worktree 与任务分支。
3. Claude 只读分析代码并处理澄清循环。
4. Claude 生成 `ExecutionPlan`。
5. 用户批准并冻结计划。
6. Codex 执行或返修。
7. Workloop 运行确定性验证。
8. 独立 Claude 会话审核。
9. 根据审核处置进入交付、返修、重新规划或阻塞。
10. 审核通过后生成 `DeliveryReport` 和任务提交。
11. 检查目标分支、必要时重新整合并重新验证审核。
12. 用户确认后完成交付。

执行与审核自动循环最多三轮。重新规划会产生新的计划版本，并重置需要重新确认的执行契约；它不应被当成普通代码返修轮次。

### 6. Agent Runtime 契约

核心不直接拼装供应商命令，而是通过统一 `AgentRuntime` 完成启动、恢复、取消和健康检查。

统一请求需要表达：任务与角色、指令、工作区、访问级别、允许命令、禁止能力、输出 Schema、runtime、model、session、总超时、无事件超时和费用预算。

统一结果需要表达：成功状态、session 引用、最终消息、标准化事件、使用量、实际运行身份、错误类型和错误详情。

统一事件至少覆盖：会话开始、消息增量、工具开始、工具完成、使用量更新、心跳、完成、失败和取消。

供应商 session 只作为连续性优化。恢复任务所需的计划、当前工作区、审核记录、验证证据和有效配置必须全部由 Workloop 落盘。

### 7. Claude Code 适配器

- 规划角色运行在任务 worktree，权限为只读或原生 plan 模式。
- 审核角色使用与规划不同的新会话，权限为只读。
- 审核会话可以运行项目策略允许的测试与检查，但不能编辑文件。
- 多轮审核可以恢复同一审核会话，以跟踪问题是否解决。
- 提示通过标准输入或供应商流式输入协议传递，不进入命令行参数。
- 使用供应商原生结构化输出能力表达计划和审核结果。
- 显式约束工具、权限、设置来源和可访问目录，不能依赖提示词声明安全边界。

### 8. Codex CLI 适配器

- 执行角色以任务 worktree 为工作根，使用 workspace-write 沙箱。
- 指令通过标准输入传递，并消费 JSONL 事件流。
- 保存可恢复 session 引用；返修优先恢复原 session。
- 通过原生 Schema 约束最终 `ExecutionResult`，但文件变化由工作区快照计算，不从最终消息解析。
- 审批策略设置为非交互运行；任何需要扩大权限的动作由 Workloop 预先暂停并向用户请求授权。
- 不使用绕过沙箱与审批的危险模式。
- 原生 code review 命令不是第一版审核主路径；审核由独立 Claude Runtime 承担。

### 9. 模型与运行时选择

- `runtime` 和 `model` 是不同配置维度。
- 第一版提供 Claude Code 与 Codex CLI 专用 Runtime，不提供任意可写命令模板。
- 任务创建后冻结 planner、executor、reviewer 的 runtime、model、CLI 版本和有效权限。
- 瞬时错误可以按错误类别有限重试，但不能静默切换 runtime 或 model。
- 切换需要人工确认，并产生新的运行身份记录；必要时重新规划或重新审核。
- Claude 规划和 Claude 审核可以使用相同模型，但必须使用隔离会话。
- executor 与 reviewer 的隔离策略至少要求不同 Agent Runtime；未来可以配置更严格的 provider 或 model family 隔离。
- API Runtime 作为架构扩展点保留，不进入第一版实现。

### 10. 项目配置与权限

- 每个项目使用版本控制内的项目配置声明默认目标分支、验证命令、受保护路径、超时和网络策略。
- `CLAUDE.md`、`AGENTS.md` 等代理说明可以提供项目约定，但不能扩大 Workloop 项目策略授予的权限。
- 工作区读写、命令、网络和额外目录访问由统一策略表达，再由适配器翻译成供应商参数。
- 默认禁止网络、部署、发布、系统级写入、秘密文件访问和工作区外写入。
- 安装依赖、访问外部服务或增加可写目录必须暂停并请求一次明确授权。
- 原始模型配置和项目配置不得通过 Web 任意写入工作区外路径。

### 11. Git 工作区与交付

- Git 项目是第一版主要目标；每个任务使用独立 worktree 和任务分支。
- 任务记录不可变 `base_commit`，不静默包含目标项目的脏修改。
- 非 Git 目录的快照写回仅保留为未来兼容路径，不进入第一版主验收。
- 审核通过后在任务分支形成确定提交，`DeliveryReport` 绑定该提交。
- 交付前检查目标分支是否仍兼容基线。
- 目标分支前进时，在任务环境中重新整合，完整重跑验证并重新审核最终 diff。
- 合并冲突暂停人工处理，不允许代理静默决定。
- 最终交付由用户确认，通过合并或拣选任务提交完成。

### 12. 确定性验证

- Codex 和 Claude 可以在各自阶段运行测试，但它们的陈述不是最终验证证据。
- Workloop 根据已批准计划和项目策略独立执行最终验证。
- 每条验证记录命令、工作目录、开始结束时间、退出码、标准输出、标准错误和运行环境摘要。
- 必需验证失败时禁止审核 `pass`。
- 没有自动测试的项目必须在计划中声明替代验证和风险，并由用户在计划批准时接受。
- 重新整合目标分支后必须重新运行全部最终验证。

### 13. 结构化工件

`ExecutionPlan` 至少包含：需求理解、非目标、涉及文件与符号、实施步骤、约束、逐项验收标准、必须运行的测试、风险和未决问题。

`ExecutionResult` 至少包含：已完成步骤、实际修改文件、运行测试、测试结果、计划偏差、剩余风险和建议下一步。

`ReviewResult` 至少包含：`pass | revise_code | replan | blocked` 处置、逐项验收结论、问题证据、文件与行号、修改建议、建议验证和总结。

`DeliveryReport` 至少包含：需求完成情况、验收结果、实际修改文件、关键实现说明、验证证据、最终审核结论、已知风险、人工后续步骤、任务分支和提交身份。

所有工件使用版本化 Schema。原始供应商输出保留用于诊断，但业务状态只依据成功校验后的结构化工件改变。

### 14. 错误、预算和取消

- 错误至少分类为：环境缺失、认证失败、瞬时服务错误、限流或过载、调用超时、无事件超时、预算耗尽、权限请求、工具失败、结构化输出失败、策略阻止、用户取消和未知运行时错误。
- 仅明确的瞬时错误允许自动重试；认证、策略、权限、Schema 和业务处置错误不盲目重试。
- 每个任务具有总时间预算、单次运行总超时、无事件超时、最大返修轮次和可选费用预算。
- 达到预算时任务暂停，不自动换模型或降低质量。
- 取消需要终止供应商进程树，并把未完成运行落盘为取消或中断状态。

### 15. 持久队列

- 可以创建多个任务，但第一版同一时间只运行一个 AI 或验证阶段。
- 等待澄清、计划批准、权限批准和交付确认的任务不占用执行槽。
- 队列顺序与声明时间持久化。
- 服务重启后重新计算可运行任务，不依赖内存线程注册表。
- 幂等阶段允许重跑；有外部副作用的阶段必须先检查已存在结果和目标状态。

### 16. Web 控制台

- Web 是主要操作入口，负责项目选择、需求提交、澄清、计划批准、进度观察、审核查看和交付确认。
- 首页优先展示正在运行、等待人工、失败、阻塞和待交付任务，而不是空白欢迎状态。
- 任务详情围绕状态机展示唯一主动作，并提供计划、变更、验证、审核和运行事件视图。
- 状态阻塞必须提供定向解决入口；需求澄清、权限请求、运行时不可用和计划重做不能共享同一个通用答复框。
- 模型页面收缩为运行时健康检查和角色 Profile 管理，不编辑任意命令模板。
- 流程页面移除任意节点排序，改为展示固定流程和各角色有效绑定。
- 第一版经验记忆不出现在主导航核心路径。
- 页面必须支持窄屏任务浏览、键盘操作、明确焦点和足够文本对比度。

### 17. 可观测性与数据保留

- 每次 AgentRun 保存标准化事件和供应商原始事件引用。
- UI 展示当前阶段、最近事件、已用时间、调用次数、预算和无事件时长。
- 提示词、回复、工具调用和命令输出根据项目策略进行脱敏。
- 密钥、令牌、环境变量值和秘密文件内容不得进入普通工件。
- 原始供应商事件支持项目级保留期限；结构化任务工件长期保留。
- 经验记忆不直接消费原始会话记录，第一版不执行跨任务自动注入。

### 18. 兼容与迁移

- 现有任务作为只读历史记录保留，不要求旧任务继续进入新工作流。
- 旧绝对路径工件引用需要容错读取或一次性迁移，单个缺失工件不能让整个任务详情失败。
- 旧模型配置可以迁移为 AgentProfile，但任意命令模板不自动获得可写 executor 权限。
- 现有 fake 后端演进为可脚本化 FakeRuntime，用于完整状态机测试。

## Testing Decisions

### 1. 最高测试缝

主要测试缝位于“持久工作流状态机通过 `AgentRuntime` 驱动任务”这一边界。

测试使用可脚本化 FakeRuntime 提供 Claude 与 Codex 的事件、结构化结果、错误和 session 行为，并使用真实临时 Git 仓库与 worktree。通过 Web API 或核心应用服务创建需求后，测试观察任务状态、Git 结果、验证证据和交付报告，而不检查内部函数调用顺序。

这个高层测试缝覆盖用户真正依赖的行为：计划门禁、执行、验证、审核处置、返修、恢复和交付。它替代为每个内部辅助函数建立大量脆弱 mock 的做法。

### 2. Runtime 协议一致性

- Claude Code 与 Codex CLI 适配器各自维护协议一致性测试。
- 测试使用本地受控假进程或 fixture JSONL，不发起真实付费模型调用。
- 覆盖标准输入、工作目录、权限参数、Schema、JSONL 事件、session 提取、恢复、取消、进程树终止、总超时和无事件超时。
- 适配器升级 CLI 版本时必须用新 fixture 验证事件解析。
- 健康检查测试验证命令存在、版本解析和认证状态映射，但不要求网络调用。

### 3. 状态机行为

- 每个允许迁移具有外部行为测试。
- 非法迁移必须被拒绝且不产生部分工件。
- 阶段结果提交失败时任务保持可恢复状态。
- 重启扫描正确标记残留运行并提供恢复、重跑或终止选择。
- 相同阶段重复提交不会重复产生不可逆副作用。
- 等待人工的任务释放执行槽，队列中的下一个可运行任务能够继续。

### 4. 计划与审核分支

- 未决问题阻止计划批准。
- 计划批准后产生不可变版本。
- `revise_code` 恢复 Codex session 并增加返修轮次。
- session 丢失时新运行可以仅凭落盘工件继续。
- `replan` 返回规划阶段并要求新的人工批准。
- `blocked` 不自动继续。
- `pass` 只有在全部必需验证成功后才能进入待交付。
- 三轮未通过后任务暂停。

### 5. Git 与工作区

- 脏工作区默认阻止创建任务。
- 任务 worktree 基于记录的 `base_commit`。
- Codex 只能改变任务 worktree。
- 工作区外写入和受保护路径变化被策略阻止。
- 快照与 diff 覆盖新增、修改、删除和重命名场景。
- 目标分支前进会使旧审核失效并触发重新整合路径。
- 冲突暂停人工处理。
- 成功交付产生可验证的任务提交并保持真实项目无额外变化。

### 6. 确定性验证

- 验证命令严格来自冻结计划和项目允许集合。
- 成功、失败、超时、无输出和大量输出均产生完整证据。
- 必需验证失败阻止 `pass`。
- 重新整合后旧验证不能复用。
- 敏感输出脱敏后仍保留足够诊断信息。

### 7. Web 行为

- 端到端 Web 测试覆盖创建任务、回答澄清、批准计划、观察执行、查看审核、授权权限和确认交付。
- 页面根据状态只展示合法主动作。
- 任务队列、运行进度和恢复状态在刷新后保持一致。
- 旧损坏任务显示部分历史不可用，而不是整个详情失败。
- 桌面和窄屏视口均验证无横向内容丢失、控件可操作和状态文本可见。
- 键盘焦点、对话框标签和语义状态不是纯视觉测试的替代项，需要静态或自动化可访问性检查。

### 8. 真实端到端验收

第一版发布门槛是使用 Workloop 修改 Workloop 自己：

1. 选择一个需要跨多个文件并新增或修改测试的真实需求。
2. Claude 直接分析任务 worktree 并生成结构化计划。
3. 用户批准计划。
4. Codex 完成实现。
5. Workloop 独立运行测试。
6. Claude 至少提出一次有效 `revise_code` 问题。
7. Codex 恢复会话并修复。
8. 服务在流程中主动重启，并从可靠节点恢复。
9. Claude 最终审核通过。
10. Workloop 生成 `DeliveryReport` 和可审计提交。
11. 用户确认并通过 Git 完成交付。

验收关注独立测试通过率、有效审核缺陷、返修成功率、人工介入次数、运行时间、调用费用、中断恢复成功率和无关代码改动量。

## Out of Scope

- 任意拖拽、增删和排序的通用工作流编辑器。
- 第一版中的 Anthropic API、OpenAI API 或其他远程模型 API Runtime。
- Workloop 自己实现完整通用模型工具调用循环。
- 允许承担可写 executor 的任意命令模板。
- 自动根据任务复杂度切换模型。
- 超时、限额或审核失败后静默切换供应商。
- 同时执行多个 AI 任务。
- 自动提炼并注入跨任务经验记忆。
- 云端任务执行、远程 worker 和团队多租户。
- 自动部署、发布、数据库迁移或生产环境操作。
- 非 Git 项目的完整交付体验。
- 自动解决 Git 合并冲突。
- 在 Web 页面编辑底层命令、全局用户配置或工作区外配置文件。
- 把供应商隐藏推理过程作为任务恢复依赖。
- 以 AI 自述代替 Workloop 的确定性验证。

## Further Notes

### 1. 核心设计原则

Workloop 的可靠性来源是：任务状态、冻结工件、真实工作区、确定性验证和人工门禁。供应商 session、模型推理和最终文本只能增强效率，不能成为唯一事实来源。

### 2. CLI 与 API 的长期关系

第一版利用 Claude Code 与 Codex CLI 已有的代码代理能力，避免重新实现搜索、编辑、Shell 和会话循环。核心契约仍然面向 Agent Runtime，而不是具体 CLI 命令；未来只有在真实评测证明 CLI 启动、稳定性、并发、成本或远程执行成为瓶颈时，才增加 API Runtime。

API Runtime 应作为新适配器加入，而不是迫使状态机或业务工件改变。加入前需要具备统一文件、Shell、测试和权限网关，否则它不能承担真实编码 executor。

### 3. 模型评测

模型选择不凭主观感觉自动化。闭环稳定后，用同一批真实任务比较固定 CLI Profile、不同 CLI 模型和未来 API Runtime，测量独立测试通过率、审核缺陷逃逸率、返修成功率、人工介入、时间、费用、恢复率和无关 diff。评测结果再决定是否增加模型升级或路由策略。

### 4. 实施方向

这是跨多个会话的大型重构，不应直接以一个实现任务推进。下一步应把本规格拆成带依赖关系的 tracer-bullet tickets，优先建立最小持久状态机与 FakeRuntime 端到端路径，再依次接入 Git worktree、Codex Runtime、Claude Runtime、验证、恢复、交付和 Web 控制台。
